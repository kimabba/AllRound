DB 권한 재현성 — grant를 마이그레이션에 codify할 것인가

작성: 2026-07-24 · security-engineer 설계 소환 · 상태: 검토안 (마이그레이션 미작성·미적용, Commander 게이트 대기)
근거: 프로덕션(bsjdgwmveokanclqwtvx)·로컬(CLI 2.108.0) 실측 쿼리, supabase/qa/platform_grants.sql, pgTAP 11파일 129 assertion, docs/rules/DATABASE_RULES.md, Supabase 공식 changelog(#45329)

⚠️ 검토자(드론) 사후 검증 — 이 문서의 근거 ① 은 사실과 다르다. 결론은 유지된다.
  • 틀린 부분: "로컬(CLI 2.108.0)은 API 롤 자동부여가 제거돼 있다"는 서술은 뭉뚱그린 것이다. pg_default_acl 을 defaclrole 별로 분리해 실측하면 로컬·프로덕션의 supabase_admin 기본권한은 완전히 동일하다. 차이는 오직 FOR ROLE postgres 한 곳이다 — 그리고 마이그레이션 객체는 postgres 가 만들기 때문에 바로 이것이 재현성 구멍의 정확한 위치다.
  • 실측(2026-07-24): FOR ROLE postgres · public 스키마 → 프로덕션은 API 롤에 테이블 DML 전체 + 함수 EXECUTE 부여. 로컬은 테이블 MAINTAIN/REFERENCES/TRIGGER/TRUNCATE 뿐(DML 없음), 함수 EXECUTE 없음. 클린 재생 직후 authenticated 는 users SELECT·match_entries INSERT 둘 다 없음으로 확인.
  • 근거 ②(플랫폼 방향)는 확인됨 — 다만 이 문서가 빠뜨린 마감이 있다. 공식 discussion #45329 원문: 신규 프로젝트 기본값 2026-05-30, 기존 프로젝트 전체 적용 2026-10-30. 공식 opt-in SQL 은 alter default privileges for role postgres in schema public revoke select, insert, update, delete on tables from anon, authenticated, service_role; 로, 위에서 측정한 차이와 정확히 일치한다(로컬 이미지가 이미 새 기본값을 탑재).
  • 그래서 달라지는 것: 2026-10-30 이후 프로덕션에서 새로 만드는 테이블·함수는 자동 권한이 붙지 않는다 (기존 객체는 유지). "새 객체를 만드는 마이그레이션이 그 자리에서 grant 를 명시한다"는 규칙은 선택이 아니라 그 날짜 전까지 갖춰야 하는 필수다. 안 갖추면 그 이후 배포되는 신규 기능이 프로덕션에서 조용히 막힌다.

1. 한 줄 결론

codify 한다. 단 "전부 열기(blanket grant)"가 아니라, 프로덕션의 현재 권한을 쿼리로 그대로 뽑아낸 스냅샷 마이그레이션(발급 명부 방식) 하나를 체인 끝에 추가하고, 이후의 새 테이블·함수는 만드는 마이그레이션 안에서 명시적으로 grant 하는 규칙 + 가드 테스트로 지킨다.

결정타: Supabase 자신이 2026-05-30부터 신규 프로젝트의 자동 권한 부여를 껐다(공식 발표, 아래 근거 ②). "롤 권한은 플랫폼이 알아서 주는 baseline이니 마이그레이션 밖"이라는 주장은 이 시점에 이미 무너졌다 — 새로 만드는 모든 환경(로컬·CI·스테이징·신규 프로젝트)은 명시 grant 없이는 앱이 통째로 막힌다. 실측으로도 그랬다.

2. 권고안 + 구조도

권고안 3줄:
  1. 스냅샷 codify — 프로덕션 실측 ACL(테이블 42 DML, 시퀀스, service_role 182 + anon 137 + authenticated 165 함수 EXECUTE)을 생성 스크립트로 뽑아 마이그레이션 1개로 커밋. 손 편집 금지.
  2. 이후 규칙 — 새 테이블·함수를 만드는 마이그레이션은 같은 파일 안에서 grant/revoke를 명시 (이미 20260716211500_club_inquiries.sql이 이 패턴). pgTAP 가드 테스트로 드리프트 차단.

3. 근거 (전부 이번 검토에서 실측·재검증)

① 같은 "기본값"이 환경마다 다르다 — 재현성 구멍의 정체.
프로덕션의 pg_default_acl(postgres 롤·public 스키마)은 새 테이블에 전체 DML, 새 함수에 EXECUTE를 anon/authenticated/service_role에 자동 부여한다. 반면 로컬(CLI 2.108.0)의 같은 항목은 그 권한이 제거돼 있다 (테이블은 TRUNCATE류만, 함수는 아예 없음 — 직접 조회로 확인). 이 차이를 벗긴 저장소 마이그레이션은 0건 — 즉 플랫폼 자체가 바뀐 것. 옛날에 지은 건물(프로덕션)은 입주만 하면 전 층 마스터키를 자동으로 주는 규약이었고, 요즘 건물(로컬·신규 환경)은 방마다 열쇠를 따로 신청해야 한다. 우리 설계도(마이그레이션)에는 열쇠 지급 기록이 한 줄도 없어서, 설계도만으로 새 건물을 지으면 아무도 문을 못 연다.
② Supabase 공식 방향: 자동 노출 폐기.
Supabase changelog #45329 "Breaking Change: Tables not exposed to Data and GraphQL API automatically" — 2026-04부터 프로젝트 생성 화면에 옵션, 2026-05-30부터 신규 프로젝트 기본값. 기존 프로젝트가 이 동작으로 갈아탈 opt-in SQL도 공식 제공. 명시 grant는 우리가 발명하는 규칙이 아니라 플랫폼의 공식 표준이 됐다.
③ 실측 수치 재검증 (2026-07-24, 프로덕션 직접 쿼리).
테이블 42개(RLS 꺼진 테이블 0) · authenticated INSERT 없는 테이블은 정확히 club_inquiry_threads/club_inquiry_messages 2개 · 함수 182개: service_role 182 / authenticated 165 / anon 137 EXECUTE. 사전 조사 수치와 완전 일치.
④ blanket이 아니라 allowlist(스냅샷)여야 하는 이유 — 실험으로 증명됨.
"전부 grant" 방식은 pgTAP 보안 테스트를 실측으로 악화시켰다(005: 14→12, 006: 8→7, 007: 6→3). anon 137/authenticated 165라는 숫자는 마이그레이션들이 일부러 걷어낸 결과이고, blanket은 그걸 되살린다. "전 직원 출입 허용 + 금지자 명단" 방식은 명단 한 줄 빠지면 뚫린다. "출입증 발급 명부" 방식은 명부에 빠지면 못 들어올 뿐이다 — 실수가 나도 안전한 쪽(문 잠김)으로 실패한다.
⑤ 이건 이론상 리스크가 아니라 로드맵에 걸린 지뢰다.
pg_dump 백업 복원은 권한(ACL)을 포함하므로 백업 기반 DR은 괜찮다. 막히는 건 "마이그레이션으로 재구축" 경로 — CI 클린 재생(이미 실제로 막혀서 platform_grants.sql 임시조치 중), 그리고 DATABASE_RULES.md 향후 과제에 명시된 스테이징 환경 신설(JY-57). 스테이징을 만드는 날 그대로 밟는다.

4. 전문가가 고려한 것

4-1. 순서 역전 문제 (핵심 난제의 해법)

체인 끝에 "grant all" 같은 변경분(델타)을 넣으면 앞선 마이그레이션의 의도적 revoke를 되살린다 — 이게 실패한 실험의 원인. 해법은 델타가 아니라 상태의 사본(스냅샷)을 넣는 것이다. 스냅샷은 "프로덕션이 지금 갖고 있는 권한 목록 그 자체"라서, 체인 어디에 있든 최종 결과는 프로덕션과 동일해진다. revoke된 짝(예: anon × 트리거 함수)은 목록에 애초에 없으므로 되살아날 수 없다. 순서가 꼬인 가계부를 거래 하나하나 다시 맞추는 대신, "오늘 잔고" 한 줄을 확정해 놓고 이후부터 다시 적는 것.
유일한 전제: 스냅샷 생성 시점과 적용 시점 사이에 다른 마이그레이션이 끼어들지 않아야 한다(짧은 freeze, §6 절차에 반영).

4-2. 미래 테이블·함수 — ALTER DEFAULT PRIVILEGES는 답인가?

아니다. 매 마이그레이션 명시 grant가 답이다. 이유:
  • default privileges는 "객체를 만든 롤" 기준으로 동작한다. 프로덕션 실측: 테이블 42개 전부 postgres 소유, postgres 롤의 default ACL이 살아 있어 db push로 만든 새 객체는 자동 grant된다(동작 확인됨). 즉 프로덕션에서는 "통한다" — 그런데 그게 바로 문제다.
  • 이 자동 grant는 새 함수 = anon 자동 노출이라는 사고 제조기다. 실제 전례: respond_club_event가 anon에 노출돼 있다가 20260719061745에서 뒤늦게 걷어냈다. 열리는 쪽으로 실패하는(fail-open) 기본값이다.
  • Supabase 신규 기본값(근거 ②)은 이 자동 grant가 없는 세상이다. default privileges에 기대는 설계는 환경마다 결과가 갈라지는 원인을 하나 더 만드는 셈.
그래서 규칙은 하나로 통일한다: 객체를 만드는 마이그레이션이 그 자리에서 권한도 선언한다. 프로덕션의 자동 grant는 2단계(§6-6, defacl 하드닝)에서 끄는 것을 권고 — 그래야 명시 grant가 유일한 경로가 되어 프로덕션과 클린 환경이 같은 방식으로 동작한다.

4-3. 프로덕션 no-op 보장

grant/revoke는 멱등(같은 걸 두 번 줘도 변화 없음)이므로, 스냅샷이 프로덕션과 정확히 같다면 적용은 정의상 no-op이다. "같다"를 믿음이 아니라 쿼리로 증명한다: 아래 권한 지문(fingerprint) 쿼리를 ⓐ 클린 재생 DB(스냅샷 포함 재생) ⓑ 프로덕션에서 각각 돌려 diff가 0인 것을 push 전에 확인하고, push 후 프로덕션 before/after diff가 0인 것으로 마무리 확인한다(쿼리는 §6). 이삿짐을 옮기기 전과 후에 같은 재고 목록을 뽑아 두 장을 겹쳐보는 것 — 한 줄이라도 어긋나면 중단.

4-4. RLS 의존 모델을 계속 따라도 되나 (질문 5)

따라도 된다 — 이 프로젝트의 전제가 실측으로 성립하기 때문. "테이블 권한은 넓게 + RLS가 행 단위 통제" 모델의 성립 조건은 모든 노출 테이블에 RLS와 정책이 있는 것인데, 42개 전부 RLS 활성 + 정책 보유(끊긴 곳 0)를 확인했다. 저장소 규칙("강제는 RLS에", 앱 쓰기 다수가 PostgREST 직행)과도 일치한다. 테이블 권한 자체를 좁히는 리모델링은 20+개 직행 경로를 전부 Edge로 옮기는 대수술이라 지금 범위가 아니며, 보안 이득도 RLS가 이미 제공 중이다.
단, 예외 패턴은 유지·확장한다: 쓰기를 서버 경로로만 열고 싶은 테이블은 club_inquiry처럼 테이블 권한 자체를 좁히는 방식(REVOKE 쓰기 + GRANT SELECT)이 맞고, 이미 그 마이그레이션 안에 codify돼 있다. 모든 방문을 열어두되 방 안 금고(RLS)가 지키는 게 기본 모델이고, 몇 개 방(club_inquiry)은 아예 문을 잠그고 직원(Edge)만 드나드는 것.

4-5. 미묘점: 함수의 PUBLIC 기본 EXECUTE

Postgres에서 ACL이 비어 있는(NULL) 함수는 내장 기본으로 PUBLIC(모든 롤)에 EXECUTE가 있는 상태다. 스냅샷 생성기는 반드시 coalesce(acl, acldefault(...))로 이 암묵 기본까지 펼쳐서 비교·생성해야 하며(§6 쿼리에 반영), PUBLIC(grantee 0) 부여분도 캡처해야 한다. 이걸 빼먹으면 지문 대조가 거짓 통과/거짓 경보를 낸다.

5. 조심할 것 — 권한이 넓어지는 구체적 실패 시나리오

S1. freeze 위반 — revoke 부활. 스냅샷을 뽑은 뒤, 적용 전에 다른 PR이 "함수 X를 anon에서 revoke"를 프로덕션에 먼저 적용하면, 스냅샷에는 X의 anon EXECUTE가 아직 들어 있으므로 나중에 적용되는 스냅샷이 그 revoke를 되살린다. 방어: 스냅샷 생성→프로덕션 적용까지 마이그레이션 freeze + 적용 직전 지문 재대조(§6-5). 어긋나면 무조건 재생성.
S2. 스냅샷 손 편집. 생성된 파일을 "보기 좋게" 정리하다 grant 한 줄을 잘못 옮기거나, 예외(club_inquiry) 줄을 지우면 그 자리에서 쓰기 권한이 열린다. 방어: 생성 스크립트 산출물을 그대로 커밋하고, 리뷰는 줄 읽기가 아니라 지문 diff = 0으로 한다.
S3. defacl 하드닝 없이 방치(옵션 B 선택 시 잔존 리스크). 이후 만드는 모든 새 함수가 프로덕션에서만 anon EXECUTE를 자동 획득한다(클린 환경·CI에는 없음). CI는 통과하는데 프로덕션만 열려 있는, 테스트가 원리적으로 못 잡는 드리프트 — respond_club_event 사고의 재발 경로다.
참고(반대 방향 실패): 새 마이그레이션에서 grant를 빼먹는 실수는 권한이 좁아지는 쪽 — CI 클린 재생·기능 테스트에서 "permission denied"로 먼저 터지고, 성격도 데이터 노출이 아니라 가용성 문제다. 이 설계는 실수의 방향을 "안전한 쪽으로 실패"로 바꾸는 설계다.

6. 적용 절차 (각 단계: 검증 → 롤백)

공통 도구 ①: 권한 지문(fingerprint) 쿼리

-- 클린재생 DB와 프로덕션에서 각각 실행해 결과 파일을 diff 한다 (diff = 0 이 통과 기준)
select kind, obj, who, priv from (
  select 'table' as kind, c.oid::regclass::text as obj,
         case a.grantee when 0 then 'PUBLIC' else a.grantee::regrole::text end as who,
         a.privilege_type as priv
  from pg_class c join pg_namespace n on n.oid = c.relnamespace
  cross join lateral aclexplode(coalesce(c.relacl, acldefault('r', c.relowner))) a
  where n.nspname = 'public' and c.relkind in ('r','v','m','S')
  union all
  select 'function', p.oid::regprocedure::text,
         case a.grantee when 0 then 'PUBLIC' else a.grantee::regrole::text end,
         a.privilege_type
  from pg_proc p join pg_namespace n on n.oid = p.pronamespace
  cross join lateral aclexplode(coalesce(p.proacl, acldefault('f', p.proowner))) a
  where n.nspname = 'public'
) x
where who in ('PUBLIC','anon','authenticated','service_role')
order by 1, 2, 3, 4;

공통 도구 ②: 스냅샷 생성 쿼리 (프로덕션 read-only, 산출물이 곧 마이그레이션 본문)

-- 테이블·시퀀스 grant 문 생성
select format('grant %s on %s to %s;',
              string_agg(distinct a.privilege_type, ', ' order by a.privilege_type),
              c.oid::regclass, a.grantee::regrole)
from pg_class c join pg_namespace n on n.oid = c.relnamespace
cross join lateral aclexplode(coalesce(c.relacl, acldefault('r', c.relowner))) a
where n.nspname = 'public' and c.relkind in ('r','S')
  and a.grantee::regrole::text in ('anon','authenticated','service_role')
group by c.oid, a.grantee
order by c.oid::regclass::text, a.grantee::regrole::text;

-- 함수 EXECUTE grant 문 생성 (PUBLIC 부여분은 grantee 0 으로 별도 확인)
select format('grant execute on function %s to %s;',
              p.oid::regprocedure, a.grantee::regrole)
from pg_proc p join pg_namespace n on n.oid = p.pronamespace
cross join lateral aclexplode(coalesce(p.proacl, acldefault('f', p.proowner))) a
where n.nspname = 'public' and a.privilege_type = 'EXECUTE'
  and a.grantee::regrole::text in ('anon','authenticated','service_role')
order by p.oid::regprocedure::text, a.grantee::regrole::text;

※ 생성물 앞에 grant usage on schema public to anon, authenticated, service_role; 1줄을 포함한다 (스키마 USAGE 없이는 테이블 grant가 무의미). revoke 문은 넣지 않는다 — allowlist라 필요 없다.

club_inquiry 예외의 표현과 드리프트 방지 (질문 6)

예외는 이미 올바른 형태로 codify돼 있다 — 20260716211500_club_inquiries.sql이 생성과 같은 파일에서 REVOKE 쓰기 + GRANT SELECT를 선언한다. 이 패턴을 예외의 표준 표기로 확정하고, 스냅샷은 (프로덕션을 그대로 읽으므로) 자동으로 이 예외를 보존한다. 앞으로 예외가 늘 때의 드리프트 방지는 4단계 가드 ⓐ의 예외 allowlist를 테스트 파일 안에 명시하는 것으로 해결한다 — 예외를 추가하려면 마이그레이션과 테스트 두 곳을 같이 고쳐야 하므로, 한쪽만 바뀐 상태가 CI에서 걸린다.

7. Commander 결정 사항

내용얻는 것대가·잔존 리스크
A. 전체안 권고 스냅샷 codify + CI 부트스트랩 은퇴 + 가드 테스트 + (2단계) defacl 하드닝 마이그레이션만으로 동작 DB 재현(스테이징·DR·CI) · 프로덕션과 클린 환경의 권한 모델 통일 · "새 함수 anon 자동 노출" 사고 경로 폐쇄 · CI가 진짜 재현성 게이트가 됨 스냅샷 절차(freeze 포함) 1회 + 이후 모든 마이그레이션에 grant 명시 습관. 하드닝 후 grant를 빼먹으면 프로덕션 배포에서 해당 기능이 막힘 — 단 CI가 먼저 잡고, 실패 방향이 노출이 아니라 잠김
B. 절반안 스냅샷 codify + CI 전환까지만, defacl 하드닝 보류 재현성 문제는 해결 · 프로덕션 변경 최소(no-op push 1회) S3 잔존: 새 함수마다 프로덕션에서만 anon EXECUTE 자동 부여 — CI가 원리적으로 못 잡는 프로덕션 전용 노출. 지금까지 실제 사고 유형(respond_club_event)이 이 경로
C. 현상 유지 codify 안 함 — 문서화 + 환경별 부트스트랩(platform_grants.sql 류) 유지 당장 할 일 없음 스테이징 신설(JY-57)·신규 환경마다 수동 부트스트랩 필요, 빠뜨리면 전면 장애 · 플랫폼 공식 방향(자동 grant 폐기)과 역행 · CI 부트스트랩이 미래 마이그레이션의 grant 누락을 계속 은폐
별도 확인 필요로 남긴 것:
  • Supabase 공식 opt-in SQL의 정확한 문안 — changelog 토론 #45329에서 적용 시점의 최신 버전을 그대로 가져와 사용 (여기 옮겨 적지 않는 이유: 문안이 갱신될 수 있어 원문 참조가 안전).
  • 지문 대조에서 뷰/materialized view가 프로덕션에 존재한다면 클린 재생과의 ACL 일치 여부 — 지문 쿼리가 relkind v·m을 포함하므로 자동으로 드러남.

이 문서는 설계 검토안이다. 마이그레이션 작성·적용·배포는 Commander 승인 후 별도 작업으로 진행한다.
