진행률 계산 중… 체크 상태는 이 브라우저에만 저장됩니다(공유 안 됨).
한 줄 요약: 데이터가 없어서 지금은 랭킹을 못 만든다. 광주협회 배점 규정은 원문으로 다 확보됐으니, 남은 병목은 "검증된 경기 결과"를 어디서 받느냐 — 광주협회 제휴다.
0단계(자기신고 점수 잠금)는 구현·검증 완료. 1단계부터는 협회 제휴가 선행 조건이라 코드보다 협의(BD)가 먼저다. 이 문서는 그 전체를 하나로 묶어 나중에 항목별로 대조·체크하기 위한 것이다.

1. 지금까지 확정된 것 (검토·리서치 결과)

질문결론상태
랭킹을 지금 만들 수 있나아니오. 검증된 경기 결과 데이터가 0. 자기신고는 조작 가능확정
데이터 소스광주협회 공식 제휴. 결과 크롤링은 이름↔계정 매칭 불가+개인정보 리스크로 제외확정
배점 규정 확보광주(2026-03-17)·KTA·KATO·전북 전문 확보. 전남 2026본은 미공개라 실패확정
점수 저장 방식사실만 저장, 점수는 규정 적용해 파생 계산. 규정이 매년 개정+오류 정정 필요하므로확정
규정 정형화 범위광주 1개 파일럿. 4단체 동시는 전남·KTFS 미확보로 재작업 확정확정
광주↔전남 통합 랭킹분리. 2026년부터 두 협회 랭킹대회 분리, 광주는 전남 우승 불인정확정(리서치가 뒤집음)
협회 간 점수 환산안 함. 공식 환산표 없음. 협회별 랭킹을 각각 따로 둠확정
정형화 판단: 포인트 계산(대회그룹 × 진출라운드 룩업)과 집계(베스트N·유효기간)는 완전 정형화 가능. 등급·부서 판정은 우승횟수+연령+선수출신+유예가 얽힌 20~40개 분기 + 협회 심의·통보 전제라 자동화 불가 → 사람/협회 개입 필수.

2. 데이터 모델 (2단계에서 생성 — 지금 만들지 않음)

기존 tennis_orgs·tennis_divisions 카탈로그(JY-120)를 참조축으로 재사용하고, 새 테이블은 3개만 추가한다.

erDiagram
    tournaments ||--o{ tournament_results : "대회의 공식 결과"
    tennis_orgs ||--o{ org_point_rules : "협회 규정(시즌 버전)"
    tennis_divisions ||--o{ tournament_results : "부서"
    users |o--o{ tournament_results : "본인 계정 연결(옵트인, NULL 허용)"
    tournament_results }o--|| ranking_snapshots : "밤 배치가 집계"

    tournament_results {
        uuid id PK
        uuid tournament_id FK
        text division_code FK
        text placement "champion|runner_up|semi|qf|r16|r32|r64|entry"
        uuid user_id FK "옵트인 매칭, NULL 허용"
        text player_name_masked "미매칭 시 마스킹"
        text source "org_official|admin"
        uuid entered_by
        timestamptz verified_at
    }
    org_point_rules {
        uuid id PK
        text org_code "gj|jn|kta|kato|jb"
        int season_year "2026"
        date effective_from "부칙 시행일"
        text source_url
        jsonb rule_config "배점표·베스트N·유효기간·타단체인정"
    }
    ranking_snapshots {
        uuid id PK
        text org_code
        int season_year
        date computed_on
        jsonb standings "파생 결과 캐시"
    }

근거 — tournament_results는 "사실"(영수증), org_point_rules는 "규칙"(시즌 버전), ranking_snapshots는 밤 배치 "결산" 캐시. rule_config를 jsonb로 두는 이유: 협회마다 규정 모양 자체가 다르다(광주 우승/비우승 2표, KTA OPEN/SEMI/REST 3표, KATO 그룹×종목, 전북 그룹제). 공통 컬럼으로 정규화하면 협회 추가마다 스키마 변경이 된다. rule_config의 구체 필드 스펙은 규정 리서치 §5.2에 이미 초안이 있다.

3. 단계별 로드맵 (체크리스트)

완료 0단계 · 자기신고 점수 잠금 (출시 블로커 아님, 데이터 0건일 때 선제 조치)

위 4항 코드는 완료. 마지막 항목(머지)만 미결. 관리자·service_role 경로는 그대로라 나중에 검증 점수 넣는 길은 열려 있음.

결정·협의 대기 1단계 · 제휴 협의 기간 (코드보다 BD가 본체)

광주 규정 원문이 정형화 가능한 수준으로 확정되고, 협회가 "결과 데이터 제공"에 원칙 합의. 여전히 점수 계산 코드는 없음.

대기 2단계 · 제휴 성사 후 · 결과 입력 인프라

관리자가 광주 대회 결과를 입력 → 규정 적용 계산이 광주 실제 1시즌 순위를 재현. 아직 유저 노출 없음(내부 검증).

대기 3단계 · 본격 랭킹 공개

광주 파일럿 랭킹이 실제 유저에게 노출되고, 개인정보·표기 정책이 다 걸려 있음. 타 협회 확장은 파일럿 성과 본 뒤.

4. Commander 결정 대기

아래는 코드가 아니라 판단이 필요한 것. 결정되면 체크.

4-1. 미결 기술 판단 — equiv_group

현재 expand_division_codes('gj_m_gold') → {gj_m_gold, jn_m_gold} 로 광주·전남 부서를 동치 취급한다(대회 출전 자격 매칭용). 2026 랭킹 분리와 겉으론 어긋나 보이나, "이 대회에 내 부서로 나갈 수 있나"(자격)와 "내 등급·랭킹을 상호 인정하나"(랭킹)는 다른 문제다. 지금 판단하지 않는다 — 협회 질문지 4번("전남은 광주 우승을 계속 인정하는가, 비대칭인가")의 답을 받은 뒤 결정. 랭킹은 어차피 협회별 분리라 자격매칭을 지금 건드릴 이유가 없다.

4-2. 지식베이스(챗봇 RAG) — 구현됨

고객이 챗봇에 "랭킹 포인트 어떻게 매겨져요?" 물으면 답하도록 rule_articles에 안내글을 넣었다. 커밋 (KB seed).
  • 기존에 광주 규정 조문 전문(제1~27조, 28글)이 이미 있었고, 제12조 배점표가 내 gj-2026.json과 정확히 일치(2026 현행본) — 서로 교차검증됨. 조문을 복제하지 않음.
  • 새로 추가: 광주 포인트 요약 · 광주↔전남 2026 분리 안내 · KTA · KATO · 전북 5글. 조문 통짜로는 답하기 어려운 질문을 사람이 읽는 요약으로 보강.
  • 모든 글에 "참고용 · 협회 공표 우선 · 앱은 점수 미산정" 프레이밍 — 설계 §5 공식 오인 리스크 방어. RAG가 이 문구까지 함께 노출.
배포 시 주의: 삽입 후 임베딩은 cron이 자동 생성한다. 프로덕션 반영 뒤 embed-pending이 돌아야 챗봇이 검색한다.
남은 KB 이슈(별건): 기존 광주 조문 28글은 (1) title에 조문 본문이 잘려 들어간 오염, (2) "명 칭·골브부·렝킹포인트" 같은 PDF 추출 띄어쓰기 노이즈가 있다. RAG 답변 품질을 떨어뜨리므로 별도 정리 대상. 이번 범위 아님.

5. 협회 질문지 (제휴 협의 때 사용)

규정 원문으로 못 푸는 12개 항목이 리서치 문서에 정리돼 있다 → tennis-ranking-point-rules.md §6. 최우선 4개:

  1. 전남 2026 랭킹규정 전문을 받을 수 있는가 (홈페이지 미게시)
  2. 광주 "베스트25"(순위)와 "베스트15"(시상)가 정말 다른 기준인가
  3. 랭킹 데이터를 API/CSV로 제공받을 수 있는가 (현재 HTML 조회만)
  4. 시·도 협회 랭킹 포인트를 앱에 표시·재가공하는 것에 대한 협회 입장/이용 조건

6. 조심할 것 (나중에 바꾸면 아픈 지점)

  • 점수만 저장하고 사실을 버리는 것 — points_earned 누적 컬럼에 의존하면 규정 개정·오류 정정 때 재계산 불가. 원천(입상 사실) 없이 랭킹을 켜는 게 가장 비싼 실수.
  • 앱 계산값을 권위 있는 값으로 제시 — 광주 규정에 "명시 안 된 사항은 협회 유권해석에 따르며 그 효력은 명문 규정과 동일"이 있다. 계산이 규정대로여도 협회와 달라질 수 있음 → "참고용, 협회 공표 우선" 표기 필수.
  • 실패 ① 자기신고 베타 랭킹 공개 → 조작 계정 1위 → 캡처 확산 → "조작 앱" 낙인. (0단계가 이걸 선제 차단)
  • 실패 ② 결과 게시판 크롤링으로 실명 랭킹 자동생성 → 미가입자 실명 노출 항의·신고 → 개인정보 + 협회 관계 동시 훼손.
  • 실패 ③ 4단체 동시 모델링 → 전남·KTFS 미확보분을 추정으로 채움 → 틀린 계산 → 전면 재작업. (광주 1개로 좁힌 이유)
