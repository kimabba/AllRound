# 올라운드 디자인 시스템 — Pureform Sports

> 확정: 2026-07-18 (kimabba). `Active Bold`를 대체하는 현재 앱 디자인 기준이다.
> 참고 방향은 Apple Sports의 정보 우선순위와 Cal.com 계열의 명료한 인터랙션이며, 특정 제품을 복제하지 않는다.
>
> **Single source of truth:** 올라운드 사용자 UI의 현재 디자인 판단은 이 문서를 따른다.
> 과거 디자인 문서와 충돌하면 이 문서가 우선하며, 실제 값은 `app/lib/theme/` 토큰으로 구현한다.

## 1. 한 문장 원칙

운동 정보를 가장 빠르게 읽고 다음 행동을 바로 고를 수 있는, 차갑고 단정한 스포츠 도구.

## 2. 시각 언어

| 역할 | 값 | 사용 |
|---|---|---|
| Canvas | `#F4F5F7` | 데스크톱 웹의 바깥 캔버스·약한 구분 영역 |
| Surface | `#FCFCFB` | 앱 Scaffold·기본 콘텐츠·목록 |
| Ink | `#161616` | 제목·핵심 숫자 |
| Muted | `#6F7176` | 보조 정보 |
| Border | `#E5E6E8` | 구분선·외곽선 |
| Cobalt | `#3156D8` | 선택·링크·주요 행동 하나 |
| Cobalt tint | `#EDF1FF` | 선택 배경 |

- 종목마다 경쟁하는 강조색을 만들지 않는다. 테니스와 풋살도 같은 코발트 계열을 사용한다.
- 그라데이션, 장식용 그림자, 유리 효과, 크고 둥근 카드, 의미 없는 배지를 사용하지 않는다.
- 정보 그룹은 카드보다 여백과 1px 구분선으로 나눈다.

## 3. 형태와 간격

- 화면 좌우 여백: 20px
- 화면 첫 콘텐츠 상단 여백: 24px
- 같은 정보 묶음 안의 간격: 8px 또는 12px
- 서로 다른 섹션 사이 간격: 24px 또는 32px
- 최소 터치 영역: 48px (iOS 44px, Android 48px 기준을 모두 충족)
- 버튼: 높이 48px, radius 6px
- 입력창: 최소 높이 48px, radius 8px
- 기본 목록 행: 최소 높이 56px
- 목록·패널: radius 8px, 꼭 필요한 경우에만 1px border
- 큰 패널 최대 radius: 12px
- 바텀시트: 상단 radius 16px
- 그림자: 기본값 없음. 모달 오버레이만 사용

간격은 `AppSpacing`, 모서리는 `AppRadius`를 사용한다. 미디어 비율, 진행률,
1~2px 광학 보정을 제외하면 화면 코드에 임의 숫자를 새로 만들지 않는다.

## 4. 타이포와 정보 위계

- Pretendard 유지
- 화면 제목과 핵심 숫자는 굵게, 자간을 조금 좁힌다.
- 대회 목록은 날짜 숫자를 가장 먼저 읽히게 하고 제목, 장소, 상태 순으로 배치한다.
- 본문은 한 화면에서 같은 굵기를 반복하지 않는다. `제목 → 핵심 메타 → 보조 메타` 3단계로 제한한다.

| 역할 | 토큰 | 기준 |
|---|---|---|
| 브랜드·큰 숫자 | `display*` | 홈의 핵심 숫자처럼 정말 필요한 한 곳만 |
| 화면 핵심 제목 | `headlineMedium` | 23px, 최대 2줄 |
| 큰 히어로 제목 | `headlineLarge` | 29px, 독립 화면에서만 |
| 섹션·카드 제목 | `titleLarge` | 17px |
| 내비·행 제목 | `titleMedium` 또는 `bodyLarge + w600` | 15~16px |
| 본문·설명 | `bodyMedium` | 14px |
| 날짜·상태·보조 정보 | `bodySmall` 또는 `labelSmall` | 11~12px |

- 같은 위계에 다른 크기나 굵기를 사용하지 않는다.
- 긴 제목은 두 줄 안에서 읽히게 하고 한 단어만 마지막 줄에 남지 않도록 문구를 다듬는다.
- 날짜, 점수, 속도처럼 비교하는 숫자는 고정 폭 숫자 표시를 우선한다.

## 5. 내비게이션

텍스트 중심 4탭과 전역 AI 진입창을 사용한다.

| 탭 | 경로 | 역할 |
|---|---|---|
| 오늘 | `/` | 맞춤 일정과 빠른 행동 |
| 대회 | `/tournaments` | 검색·달력·상세·신청 |
| 모임 | `/clubs` | 클럽 검색·가입·운영 |
| MY | `/profile` | 프로필·기록·설정 진입 |

하단 내비게이션 위의 `AI에게 물어보기` 진입창은 일반 사용자 화면에서 항상 보인다. 누르면 절반 높이 바텀시트로 열리고 전체 채팅 화면으로 확장할 수 있다. 대회·클럽 상세 정보는 사용자가 `현재 화면 연결`을 직접 선택한 경우에만 질문에 포함한다. 로그인·온보딩·스피드건·어드민 화면에는 표시하지 않는다.

룰북은 MY의 도움말 메뉴에서 `/rules`로 진입한다. 알림과 설정도 각 화면의 문맥상 액션으로 둔다.

## 6. 화면별 기준

- 홈: 날짜, 한 문장 질문, 추천 대회 목록을 첫 화면에 둔다.
- 대회: 큰 날짜 숫자와 평면 목록을 기본으로 하고 달력은 보조 탐색으로 둔다.
- 모임: 나의 모임을 먼저 보여주고, 추천 모임은 로고·이름·지역·인원·관심 행동만 남긴다. 종목명은 카드마다 반복하지 않는다.
- 모임 검색: 검색어·정렬·`지역·요일·조건` 통합 필터로 구성한다. `모집 중`은 통합 필터 안에서 선택하며 별도 바로가기 필터를 중복 배치하지 않는다.
- 룰북: 카테고리와 글을 문서 목록처럼 보이게 한다.
- MY: 프로필과 핵심 통계는 한 덩어리로, 세부 메뉴는 구분선 목록으로 둔다.
- 인증·온보딩: 제품 설명보다 입력과 다음 행동이 먼저 보이게 한다. 이메일 신규 가입은
  생년월일을 계정 생성 전에 받고, Google은 기존 회원 로그인임을 버튼과 안내에서 분명히 한다.
- 알림·관심·차단: 별도 장식 카드 대신 평면 목록, 구분선, 명확한 빈 상태를 사용한다.
- 입력 화면: 섹션 제목과 필드 순서로 위계를 만들고 카드 중첩을 피한다.

## 7. 구현 규칙

- 색과 radius는 `app/lib/theme/` 토큰을 사용한다.
- 공통 표면은 `AppCard`의 filled/outlined 변형을 우선한다.
- 화면 좌우 여백과 섹션 간격은 `AppSpacing`을 사용한다.
- 화면 안에서 재사용되는 행·헤더·입력 영역은 `app/lib/widgets/` 공통 위젯으로 올린다.
- 기능 상태를 색만으로 전달하지 않는다. 텍스트·아이콘·형태를 함께 사용한다.
- 로딩, 오류, 빈 상태에서도 레이아웃 위계가 바뀌지 않게 한다.
- 새 UI는 라이트·다크 테마, 큰 글자, 48px 터치 영역을 함께 확인한다.

## 8. 디자인 프리뷰

`USER_DESIGN_PREVIEW=true`인 웹 실행은 운영 인증·권한을 대신하지 않는 로컬 UI 점검 전용이다.

- 데스크톱 브라우저에서도 앱 본문을 최대 390px 폭으로 고정해 모바일 레이아웃을 확인한다.
- `/login`, `/onboarding`, `/notifications`, `/favorites`, `/blocked-users`, `/rules`, `/speed-gun` 등 원하는 사용자 경로를 주소에서 바로 열 수 있다.
- 홈·대회·클럽·룰북·알림·관심·차단 화면은 프리뷰 데이터를 사용해 백엔드 장애와 무관하게 상태별 디자인을 확인한다.
- 릴리스 빌드에서는 기존 `AppConfig.assertConfigured()` 가 프리뷰 플래그 유입을 차단한다.

## 9. 공통 레이아웃 계약

### 화면

- 사용자 화면의 기본 모바일 기준 폭은 390px이며 320px에서도 가로 잘림이 없어야 한다.
- 데스크톱 웹에서는 모바일 본문 폭을 유지하고 바깥 캔버스만 확장한다.
- 상단 앱바는 56px, 하단 내비게이션은 64px, 전역 AI 진입창은 60px를 기준으로 한다.
- 키보드가 열리면 하단 내비게이션과 전역 AI 진입창은 숨기고 현재 입력 행동을 우선한다.
- 고정 하단 행동이 있을 때 스크롤 콘텐츠 마지막 항목이 가려지지 않도록 안전 여백을 둔다.

### 정렬

- 화면 제목, 섹션 제목, 목록 본문, 입력창의 왼쪽 기준선은 원칙적으로 20px에 맞춘다.
- 아이콘은 텍스트 기준선에 광학적으로 맞추며 단순 수학적 중앙 정렬이 어색하면 1~2px만 보정한다.
- 구분선은 정보 계층과 같은 시작점에 맞춘다. 전체 폭 선과 들여쓴 선을 한 목록에서 섞지 않는다.

## 10. 색·표면·모서리 계약

### 색

- `surface`는 기본 콘텐츠, `surfaceContainerLow`는 약한 구분, `primaryContainer`는 선택 상태에만 사용한다.
- 코발트는 현재 선택, 링크, 주 행동 하나에만 사용한다.
- 성공·경고·오류는 의미 전달에 필요한 경우만 별도 상태색을 사용하고 반드시 텍스트나 아이콘을 함께 둔다.
- 화면 코드에서 팔레트 색상값을 직접 작성하지 않는다.

### 표면

- 카드가 없어도 여백과 구분선으로 이해되면 카드를 만들지 않는다.
- 중첩 카드, 장식용 그림자, 여러 단계의 테두리를 만들지 않는다.
- 모달과 시트만 오버레이 깊이를 사용하고 일반 목록은 평면으로 유지한다.

### 모서리

| 대상 | radius |
|---|---:|
| 작은 상태 표시 | 3px |
| 버튼 | 6px |
| 입력창·목록 패널 | 8px |
| 카드 | 10px |
| 큰 히어로 | 12px |
| 바텀시트·대형 모달 | 16px |
| 필터·상태 pill | 999px |

14px처럼 토큰에 없는 중간 radius를 화면별로 만들지 않는다. Pill은 필터나 상태처럼
형태 자체가 의미를 가질 때만 사용한다.

## 11. 아이콘과 인터랙션 계약

- 기본 아이콘은 한 화면에서 같은 Material Rounded 계열을 사용한다.
- 본문 보조 아이콘 18px, 목록 아이콘 20px, 주요 액션 22~24px를 기준으로 한다.
- 같은 행동에 화면마다 다른 아이콘을 쓰지 않는다.
- 왼쪽 의미 아이콘과 오른쪽 이동 아이콘이 같은 뜻을 반복하면 하나를 제거한다.
- 누를 수 있는 아이콘은 보이는 크기와 별개로 48px 터치 영역을 확보한다.
- 눌림은 ripple 또는 짧은 표면 변화로 표현한다. 장식적인 튕김이나 무한 애니메이션은 사용하지 않는다.
- 화면 전환·시트는 250~400ms, 작은 상태 변화는 150~200ms 범위의 `AppDuration`을 사용한다.
- 키보드 포커스와 스크린리더 레이블을 제공한다.

## 12. 공통 컴포넌트 계약

| 컴포넌트 | 시각·동작 기준 |
|---|---|
| 화면 헤더 | 56px, 왼쪽 제목, 오른쪽 문맥 행동 최대 2개 |
| 섹션 헤더 | 17px 제목, 보조 행동은 텍스트 버튼 하나 |
| 기본 버튼 | 48px, radius 6px, 한 영역의 filled 버튼은 하나 |
| 아이콘 버튼 | 최소 48×48px, radius 8px |
| 텍스트 입력 | 최소 48px, radius 8px, 포커스 선 1.5px |
| 목록 행 | 최소 56px, 한 개의 핵심 제목과 최대 두 단계 메타 |
| 필터 칩 | 선택과 해제를 글자·색·형태로 함께 표현 |
| 카드 | 독립된 정보 묶음일 때만 사용, 중첩 금지 |
| 빈 상태 | 이유, 다음 행동, 필요한 경우 CTA 하나 |
| 오류 상태 | 원인 요약, 사용자가 할 수 있는 복구 행동 |
| 로딩 상태 | 실제 콘텐츠 형태를 유지하는 스켈레톤 우선 |
| 토스트 | 한 문장 결과, 필요한 경우 복구 행동 하나 |
| 하단 내비 | 오늘·대회·모임·MY 4개, 현재 탭을 색 외 형태로도 표시 |
| 전역 AI 진입창 | 60px, 코발트 아이콘과 현재 화면 문맥 표시, 키보드·전체 채팅에서는 숨김 |

공통 위젯이 있는데 화면에서 같은 모양을 새로 만들지 않는다. 공통 위젯에 없는 상태가
필요하면 화면에 복제하기 전에 공통 컴포넌트 계약을 먼저 확장한다.

## 13. 전역 AI 채팅 기준

- 시트 헤더는 56px 안에서 드래그 핸들, 제목, 전체 화면 행동을 정리한다.
- 빈 대화 제목은 `headlineMedium`을 사용하고 콘텐츠 상단 24px에서 시작한다.
- 추천 질문은 56px 평면 행으로 만들고 한 행에는 이동 의미 아이콘을 하나만 둔다.
- 입력 영역은 본문과 1px 선으로 나누고 좌우 12px, 위아래 10~12px를 사용한다.
- 입력창은 48px에서 시작해 여러 줄 입력 시 최대 약 104px까지만 커진다.
- 전송 버튼은 48×48px, radius 8px로 입력창 하단에 맞춘다.
- 대회·클럽 문맥 연결은 기본 해제이며 사용자가 직접 선택해야 한다.
- 연결 상태는 체크, 문구, 배경을 함께 바꾸고 공개 정보만 사용한다는 설명을 유지한다.
- 전체 채팅 화면에서도 같은 추천 행, 입력창, 메시지 컴포넌트를 재사용한다.

## 14. 화면별 적용 지도

| 묶음 | 화면 | 사용할 기본 패턴 |
|---|---|---|
| 핵심 진입 | 홈 | 날짜 헤더, 핵심 질문, 평면 추천 목록, 전역 AI |
| 대회 | 목록·상세·신청 | 필터, 날짜 중심 행, 정보 섹션, 고정 주요 행동 |
| 모임 | 목록·상세·생성 | 나의 모임, 통합 검색·필터, 추천 모임, 운영 섹션, 단계형 입력 |
| MY | 프로필·기록·설정 | 프로필 요약, 핵심 수치, 평면 설정 목록 |
| AI | 전역 시트·전체 채팅 | 시트 헤더, 추천 행, 메시지, 고정 입력 영역 |
| 보조 | 알림·관심·친구 일정·룰북·차단 | 화면 헤더, 평면 목록, 명확한 빈 상태 |
| 인증 | 로그인·온보딩·비밀번호 | 짧은 설명, 입력 우선, 가입 전 연령 확인, 한 단계 한 주요 행동 |
| 전문 도구 | 스피드건 | 같은 토큰과 컨트롤, 측정 정보는 더 높은 밀도 허용 |
| 운영 도구 | 관리자 | 같은 색·타이포·상태 규칙, 데스크톱 정보 밀도 허용 |

## 15. 상태와 접근성

모든 네트워크 화면은 다음 상태를 같은 자리에서 처리한다.

1. 초기 로딩
2. 콘텐츠 있음
3. 결과 없음
4. 연결·권한 오류
5. 재시도 중
6. 비활성·권한 제한

- 텍스트 130%에서도 핵심 행동과 제목이 잘리지 않아야 한다.
- 본문과 배경은 WCAG AA 대비를 만족한다.
- 색만으로 선택, 오류, 마감 상태를 전달하지 않는다.
- 스크린리더 순서는 시각적 읽기 순서와 같아야 한다.
- 입력 오류는 해당 필드 가까이 설명하고 사용자가 수정할 방법을 말한다.

## 16. 디자인 검증 매트릭스

새 화면과 변경 화면은 최소 다음 조건을 확인한다.

| 조건 | 확인 내용 |
|---|---|
| 320×568 | 작은 기기 가로 잘림·하단 행동 겹침 |
| 390×844 | 기본 모바일 레이아웃과 수직 리듬 |
| 430×932 | 큰 모바일에서 불필요하게 퍼지는 여백 |
| 웹 프리뷰 | 모바일 본문 폭, 포인터·키보드 포커스 |
| 텍스트 130% | 제목·버튼·행 높이와 줄바꿈 |
| 라이트·다크 | 대비, 표면 단계, 상태색 |
| 키보드 열림 | 입력창, 하단 내비, 스크롤 가림 |
| 긴 한국어 데이터 | 2줄 제목, 말줄임, 메타 정렬 |
| 로딩·빈 상태·오류 | 콘텐츠 전환 시 레이아웃 점프 |

Flutter 변경 후 `flutter analyze`, `flutter test`와 주요 화면 브라우저 스크린샷 비교를
통과해야 한다. 공통 컴포넌트는 위젯 테스트에서 크기, 레이블, 상태 전환을 확인한다.
정의되지 않은 13·14·15px 등의 radius와 44·50·52·54·56px 고정 버튼 높이는
`scripts/harness/check_static_rules.py`가 차단하며 `AppRadius`·`AppSizes` 토큰을 사용한다.

## 17. 유지 규칙

1. 디자인 구조 변경은 구현 전에 간단한 설계와 기준 화면을 사용자에게 보여주고 승인받는다.
2. 승인된 결정은 이 문서에 먼저 반영하고 토큰·공통 위젯·화면 순서로 구현한다.
3. 기능 변경과 전면 디자인 변경을 한 번에 섞지 않는다.
4. 새 색, 글자 크기, 간격, radius가 필요하면 화면에 직접 추가하지 말고 토큰 변경 사유를 기록한다.
5. 변경 전후 스크린샷과 상태별 검증 결과를 PR에 남긴다.
6. 과거 디자인 문서는 삭제하지 않고 `보관 문서`로 표시해 현재 기준과 혼동되지 않게 한다.
7. 첫 기준 화면에서 톤을 확정한 뒤 핵심 동선, 보조 화면, 전문 도구 순으로 확장한다.

## 18. 적용 순서

1. 전역 AI 시트·전체 채팅·하단 내비를 첫 기준 화면으로 완성한다.
2. 화면 헤더·행·입력창·상태 컴포넌트를 공통 위젯으로 정리한다.
3. 홈 → 대회 → 모임 → MY 핵심 동선을 마이그레이션한다.
4. 알림·관심·친구 일정·룰북·차단·인증 화면을 마이그레이션한다.
5. 스피드건과 관리자 화면은 기능 밀도를 유지하며 시각 규칙만 통일한다.
6. 전체 화면 매트릭스와 실제 기기 QA를 통과한 뒤 디자인 통일 작업을 완료 처리한다.
