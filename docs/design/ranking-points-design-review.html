협회별 랭킹 포인트 시스템 — 설계 검토

작성: 2026-07-24 · backend-architect 설계 소환 · 상태: 검토안 (코드/DB 미적용, Commander 게이트 대기)
근거 자료: docs/research/tennis-grade-systems.md, docs/research/tennis-ecosystem.md, 마이그레이션 009·061·20260710(부서 카탈로그), docs/rules/DATABASE_RULES.md

1. 한 줄 결론

지금은 "랭킹"을 만들 수 없습니다. 랭킹의 원료인 "검증된 경기 결과" 데이터가 하나도 없기 때문입니다.
지금 만들 수 있는 것은 "내 대회 이력"(개인 기록장)까지이고, 진짜 랭킹은 광주협회 공식 결과 데이터 제휴가 선행 조건입니다. 다행히 이미 광주협회 이사 접점이 있어 이것이 가장 현실적인 경로입니다. 그때를 위해 지금 막아둘 것은 딱 하나 — 유저가 자기 점수(points_earned)를 스스로 쓸 수 있는 현재 구조를 랭킹에 절대 연결하지 않는 것입니다.

2. 왜 지금은 안 되는가 — 입력 데이터 문제 (질문 1)

랭킹 점수는 "누가, 어떤 대회에서, 몇 등 했나"를 알아야 계산됩니다. 현재 상황:

  • 크롤러는 대회 요강(모집 공고)만 수집합니다. 경기 결과는 수집하지 않습니다.
  • 유일한 결과 데이터인 match_entries는 유저 자기신고이고, RLS가 본인 전체 CRUD를 허용해 자기 점수를 마음대로 입력·수정할 수 있습니다.
  • 협회 공식 랭킹 API·공개 데이터셋은 없습니다 (리서치 문서로 확인됨).
이 상태로 랭킹을 공개하면 깨지는 것: 자기신고 점수로 순위가 매겨지는 순간, 순위를 올리고 싶은 유저는 우승 기록을 지어내면 됩니다. 검증 수단이 없으므로 반박도 못 합니다. 랭킹의 유일한 상품 가치는 "신뢰"인데, 첫날부터 신뢰가 0인 랭킹은 조롱거리가 되고 앱 전체 신뢰까지 깎아 먹습니다. 참가자가 스스로 채점해서 제출하는 마라톤 기록증 같은 것 — 종이는 만들 수 있지만 아무도 믿지 않습니다.

데이터 소스 옵션 비교

옵션신뢰도구현 비용법적 리스크평가
A. 자기신고 유지
(현 match_entries)		 낮음 — 조작 가능 0 (이미 있음) 낮음 (본인 데이터) "이력"으로는 유효. 랭킹 원료로는 불가.
B. 결과 페이지 크롤링
(협회 사이트 입상자 게시판)		 중 — 원문은 공식이나 이름→계정 매칭이 불가능(동명이인, 실명 미보유) 높음 — 협회 사이트가 불안정(ECONNREFUSED·SSL 오류 빈발, 리서치 §7)하고 게시 형식이 제각각 높음 — 제3자 실명+성적을 동의 없이 수집·영리 서비스에 노출. 개인정보보호법상 공개정보라도 정당한 처리 근거 필요 비권고. 매칭 실패가 기본값이고 법적 노출이 가장 큼.
C. 협회 공식 제휴
(광주협회 파일럿)		 최상 — 협회가 결과의 원천 중 — BD(제휴 협의)가 본체, 기술은 관리자 입력 화면 정도 최저 — 협회 위탁·제공 근거로 처리, 표기도 "공식 제공" 가능 권고. 광주협회 이사 접점 보유 + 광주는 전국 최초 동호인 랭킹제(1~1000위) 운영 주체라 니즈 접점이 있음
D. 앱이 대회 운영 도구가 됨
(대진표·결과 입력을 앱이 제공)		 최상 — 데이터가 앱에서 태어남 매우 높음 — 별도 제품 수준 낮음 장기 방향으로는 최선이나 지금 논할 단계 아님
권고: C (광주협회 제휴)를 본선으로, A는 "개인 이력" 기능으로 격을 나눠 유지. B(결과 크롤링)는 이름 매칭 문제와 개인정보 리스크 때문에 제외. D는 C가 성공한 뒤의 다음 카드.

3. 설계안 (질문 2·3·5) — "원천 보존 + 점수는 계산해서 파생"

3-1. 핵심 원칙 3가지

  1. 원천(입상 사실)과 점수(계산 결과)를 분리한다. 저장하는 것은 "2026 빛고을배 골드부 준우승"이라는 사실뿐. 몇 점인지는 그때그때 규정 테이블을 적용해 계산한다(파생형). 가계부에 영수증(사실)만 붙여 두면, 월말 결산(점수)은 언제든 다시 뽑을 수 있습니다. 결산 숫자만 적어두고 영수증을 버리면 규칙이 바뀌었을 때 다시 계산할 방법이 없습니다.
  2. 규정은 코드가 아니라 데이터(연도 버전)로 둔다. 협회 규정은 매년 개정된다(KTA 2024.02 개정 확인). 코드 분기로 짜면 협회 하나 추가·개정 하나 반영마다 앱 배포가 필요하고 과거 시즌 재계산이 불가능해진다.
  3. 검증된 결과만 랭킹에 들어간다. 결과 테이블은 관리자/서버(service_role)만 쓸 수 있고, 유저 자기신고는 랭킹 계산에서 원천 배제. 강제는 앱이 아니라 RLS가 한다(이 프로젝트의 기존 원칙).

3-2. 시간의존 규칙(1년 유예·베스트N·시즌 롤오버)은 왜 파생형인가 (질문 3)

저장형(누적 컬럼에 계속 더하기)은 KTA "입상 후 만 1년 경과 시 등급 복귀", KATA "베스트 15", 광주 "연간 랭킹 갱신" 같은 규칙과 근본적으로 안 맞습니다. 이 규칙들은 전부 "오늘 날짜 기준으로 과거 기록을 다시 보는" 계산이라, 어제 더해둔 누적값은 오늘 이미 틀린 값이 됩니다.

파생형(밤마다 원천에서 전체 재계산)을 권고합니다. 동호인 수 규모(광주 약 15,000명)에서는 전체 재계산이 밤 배치 1회로 충분히 가볍고, 규정이 개정되면 규정 테이블만 바꾸고 다시 돌리면 과거 시즌도 재현됩니다. 이미 이 프로젝트는 pg_cron 배치(embed-pending 등)를 운용 중이라 새로운 운영 부담이 아닙니다. 은행 잔고(저장형)가 아니라 학점 평점(파생형)입니다. 평점은 성적표 원본만 있으면 계산 규칙이 바뀌어도 언제든 다시 산출됩니다.

3-3. 제안 스키마 (미래 적용분 — 지금 만들지 않음, 3단계에서 생성)

erDiagram
    tournaments ||--o{ tournament_results : "대회의 공식 결과"
    tennis_orgs ||--o{ org_point_rules : "협회별 규정(연도 버전)"
    tennis_divisions ||--o{ tournament_results : "부서"
    users |o--o{ tournament_results : "본인 계정 연결(있을 때만)"
    tournament_results }o--|| ranking_snapshots : "밤 배치가 집계"

    tournament_results {
        uuid id PK
        uuid tournament_id FK
        text division_code FK "tennis_divisions 참조"
        text placement "winner|runner_up|semi|qf|r32"
        uuid user_id FK "옵트인 매칭, NULL 허용"
        text player_name_masked "미매칭 시 마스킹 표기"
        text source "org_official|admin"
        uuid entered_by "입력 관리자"
        timestamptz verified_at
    }
    org_point_rules {
        uuid id PK
        text org_code FK
        int season_year "2026"
        date valid_from
        date valid_to
        jsonb rule_config "배점표·베스트N·유예기간"
    }
    ranking_snapshots {
        uuid id PK
        text org_code FK
        int season_year
        date computed_on
        jsonb standings "파생 결과 캐시"
    }
  • 테이블 3개가 전부입니다. tournament_results(원천 사실), org_point_rules(연도별 규정), ranking_snapshots(밤 배치 계산 결과 캐시). 기존 tennis_orgs·tennis_divisions 카탈로그(JY-120)를 그대로 참조축으로 재사용 — 부서 사전을 또 만들지 않습니다.
  • rule_config를 jsonb로 두는 이유: 협회마다 규정의 모양 자체가 다릅니다 (KTA 페어합산+1년 유예 / KATO 그룹×부서+32강 문턱 / KATA 베스트15 / 전북 명시 점수제). 공통 컬럼으로 정규화하려는 순간 협회 하나 추가마다 스키마 변경이 됩니다. 계산기는 규정 "종류"별로 소수의 SQL/Edge 함수. 단, 파일럿(광주) 1개 규정만 우선 정형화하고 나머지 협회는 콘텐츠(rule_articles)로 유지 — 4단체 전부를 미리 모델링하는 것은 과잉설계입니다.
  • UNIQUE(tournament_id, division_code, placement, player_name)급 유니크 제약으로 중복 입력을 DB가 차단합니다.
  • 기존 match_entries는 그대로 "개인 이력"으로 살리되, points_earned는 랭킹과 무관한 자기 기록 필드로 격하(장기적으로 컬럼 제거 후보).

3-4. 협회 간 환산 (질문 4)

협회별 랭킹을 각각 따로 둡니다. 앱이 환산하지 않습니다.
공식 환산표가 존재하지 않고(리서치 확인: KTA·KATO·KATA 상호 인정도 부분적·조건부), 2025-12 KTA·KATO 통합 협약조차 KATA는 미참여입니다. 이 상황에서 앱이 임의 환산 점수를 만들면 그것이 곧 "비공식 통합 랭킹"이 되어 §5의 법적·평판 리스크를 정면으로 밟습니다.
equiv_group은 지금처럼 "출전 자격 매칭"(이 대회에 내 부서로 나갈 수 있나) 용도까지만 씁니다. 점수 환산 근거로 확장하지 않습니다. 유일한 예외는 광주↔전남 — 규정 자체를 공유하는 단일 권역이므로 하나의 랭킹풀로 묶는 것은 "환산"이 아니라 "사실 반영"입니다(단, 협회 제휴 시 양측 확인 필요). 수능 점수와 토플 점수를 한 줄로 세우는 공식 표가 없는데 학원이 임의로 만들면, 그 표가 틀렸을 때 책임은 학원이 집니다.

4. 무결성·보안 — 지금 구조의 구멍과 막는 법 (질문 5)

현재 구멍: match_entries RLS가 user_id = auth.uid() 전체 CRUD 허용 → 앱 쓰기 다수가 PostgREST 직행인 이 프로젝트에서, 유저는 API로 points_earned = 99999를 직접 INSERT/UPDATE 할 수 있습니다. 이 값이 어떤 화면·정렬·랭킹에도 쓰이지 않는 동안은 무해하지만, 랭킹에 연결되는 순간 치명적입니다.

구체적 조작 시나리오와 방어:

시나리오방어 (DB가 막는 것)
유저가 자기 points_earned를 임의 입력 랭킹은 tournament_results(admin/service_role 전용 쓰기)만 집계. 자기신고 테이블은 집계에서 원천 배제. 추가로 트리거로 source='manual'이면 points_earned=0 강제(1단계에서 즉시 가능한 최소 조치)
유령 대회 — 존재하지 않는 대회를 만들어 우승 기록 tournament_results는 status='published'(검수 통과) 대회에만 FK+CHECK로 연결. 대회 등록 자체가 어드민 검수 큐를 거치는 기존 파이프라인 재사용
중복 입력 — 같은 대회 성적을 두 번 등록 UNIQUE 제약(대회×부서×순위×선수). 앱 코드가 아니라 DB 제약이 막음
타인 사칭 — 남의 성적을 자기 계정에 연결 결과→계정 연결은 관리자 확인 또는 본인 옵트인 클레임 절차로만. 유저 셀프 연결 UPDATE는 RLS로 차단

권한 구조 요약: 쓰기 = admin(is_admin()) + service_role Edge Function만 / 읽기 = 공개 범위 정책에 따라 authenticated / 계산 = pg_cron 배치(service_role). 기존 is_admin() SECURITY DEFINER 헬퍼와 검수 큐 인프라를 그대로 재사용하므로 새 보안 표면이 거의 늘지 않습니다.

5. 제품·법적 리스크 (질문 6)

리스크내용대응
공식 랭킹 오인 앱 점수가 협회 공식 랭킹으로 오인되면 협회와의 관계 악화 + 유저 항의("앱에선 내가 3위인데 왜 승급 안 되나"). 협회명·로고 무단 사용 시 더 심각 제휴 전에는 협회명을 랭킹에 붙이지 않음. 제휴 후에도 "○○협회 제공 데이터 기반" 표기와 면책 문구를 화면에 상시 노출. 승급·출전자격 판정은 협회 원본만 유효함을 명시
실명+성적 공개 (개인정보보호법) 성적은 실명과 결합하면 개인정보. 앱 미가입자의 실명 성적을 동의 없이 노출하면 §15 처리 근거 문제. 개인정보처리방침 미비 시 과태료(SECURITY_WEB_SAAS 체크리스트 §1 해당) 앱 가입 + 옵트인한 유저만 실명/닉네임 노출. 미매칭 입상자는 마스킹("김○수") 또는 미노출. 랭킹 공개 동의를 별도 항목으로 받고 개인정보처리방침에 처리 목적 추가. 탈퇴 시 랭킹 표기 즉시 제거(보존 의무 데이터와 분리)
규정 해석 오류 리서치 문서 자체에 "추정" 표기가 많음(광주 승급 규정도 추정). 틀린 배점으로 계산하면 신뢰 훼손 파일럿 협회 규정은 반드시 협회로부터 원문 확보 후 정형화. 추정 규정으로는 랭킹 계산을 켜지 않음

6. 조심할 것 — 나중에 바꾸면 아픈 지점 (질문 7 연계)

  • 점수만 저장하고 사실을 버리는 것. points_earned류 누적 컬럼에 의존하기 시작하면 규정 개정·오류 정정 때 재계산이 불가능합니다. 원천(입상 사실) 테이블 없이 랭킹을 켜는 것이 가장 비싼 실수입니다.
  • 시즌 개념. 대회 개최일이 이미 저장되므로 시즌은 파생 가능 — 지금 컬럼을 만들 필요는 없습니다. 단, 협회 규정을 정형화할 때는 반드시 season_year·유효기간을 규정 테이블 축으로 두어야 합니다(연도 개정 대응).
  • 자기신고 점수가 화면에 노출되는 순간. "내 이력" 화면에라도 points_earned를 숫자로 보여주기 시작하면 유저는 그것을 랭킹으로 인식하고, 나중에 검증 랭킹으로 교체할 때 "내 점수가 사라졌다" 반발이 생깁니다. 노출 자체를 하지 않는 것이 싼 예방입니다.
  • 실패 시나리오 ①: 자기신고 기반 베타 랭킹 공개 → 조작 계정이 1위 → 커뮤니티에서 캡처 확산 → "조작 앱" 낙인. 복구 비용이 기능 가치보다 큼.
  • 실패 시나리오 ②: 협회 결과 게시판 크롤링으로 실명 랭킹 자동 생성 → 미가입 동호인이 자기 실명 노출에 항의·신고 → 개인정보 이슈 + 협회 관계 동시 훼손.
  • 실패 시나리오 ③: 4개 전국단체 규정을 한꺼번에 모델링 시도 → KTFS는 원문조차 미확보 상태라 추정으로 채움 → 틀린 계산 발견 → 전면 재작업. 파일럿 1개 협회로 좁혀야 하는 이유.

8. 단계별 로드맵

단계하는 것DB에 지금 넣어야 하는 것지금 안 넣어도 되는 것
0단계 · 지금~출시 (2주 로드맵 내)
출시 블로커 아님		 랭킹 기능 없음. match_entries는 "내 대회 이력"으로만 노출(점수 숫자 비노출). 광주협회 제휴 타진 시작(가상계좌 BM 접점과 같은 채널) 최소 위생 1건 — match_entries에 UNIQUE(user_id, tournament_id, division) + points_earned 자기신고 시 0 강제(트리거 또는 컬럼 노출 제거). 되돌리기 쉽고 나중의 데이터 오염을 지금 막음 결과·규정·스냅샷 테이블 전부. 시즌 컬럼. 랭킹 UI
1단계 · 제휴 협의 기간 파트너 상호확인(내 이력에 파트너가 "맞음" 표시 — 소셜 증빙, 랭킹 아님). 협회로부터 광주 랭킹 규정 원문 확보 → 정형화 검증 match_entries.partner_confirmed_at 정도의 소규모 추가 여전히 점수 계산 없음
2단계 · 제휴 성사 후 tournament_results + org_point_rules 생성(§3-3). 관리자 결과 입력 화면. 과거 1시즌 백필 테이블 2개 + RLS(admin/service_role 전용) + UNIQUE 제약 스냅샷 캐시(유저 적으면 실시간 뷰로 충분)
3단계 · 본격 랭킹 밤 배치 파생 계산 + ranking_snapshots + 랭킹 화면 공개(옵트인·마스킹 정책 적용). "협회 제공 데이터 기반" 표기 스냅샷 테이블 + pg_cron 잡 타 협회 확장·통합점수 — 파일럿 성과 본 뒤

본 문서는 설계 검토안입니다. 스키마 생성·RLS 변경·트리거 추가는 Commander 승인 후 별도 PR로 진행합니다.
